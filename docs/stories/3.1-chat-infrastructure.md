## Technical Notes

```sql
CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  creator_id UUID NOT NULL REFERENCES users(id),
  consumer_id UUID NOT NULL REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(creator_id, consumer_id)
);

CREATE TABLE messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  sender_id UUID NOT NULL REFERENCES users(id),
  content TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ NOT NULL
);

CREATE INDEX idx_messages_conversation ON messages(conversation_id, created_at DESC);
CREATE INDEX idx_messages_expires_at ON messages(expires_at);
```

```typescript
// SSE implementation
// app/api/messages/stream/route.ts
export async function GET(request: Request) {
  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      // Subscribe to Supabase Realtime
      const channel = supabase
        .channel('messages')
        .on('postgres_changes', { event: 'INSERT', schema: 'public', table: 'messages' }, (payload) => {
          controller.enqueue(encoder.encode(`data: ${JSON.stringify(payload.new)}\n\n`));
        })
        .subscribe();
    },
  });

  return new Response(stream, {
    headers: { 'Content-Type': 'text/event-stream' },
  });
}
```

## Dev Agent Record

### Implementation Summary
- Created database schema for conversations and messages with 24-hour expiration
- Implemented REST API endpoints for creating and retrieving messages
- Built Server-Sent Events (SSE) endpoint for real-time message delivery using Supabase Realtime
- Created cleanup endpoint for deleting expired messages (cron job)
- Configured RLS policies for secure access control
- Enabled Supabase Realtime for messages table

### Technical Decisions
- Used SSE instead of WebSockets for simpler implementation and better HTTP/2 compatibility
- Implemented 24-hour message expiration (Snapchat-style ephemeral messaging)
- Conversations use unique constraint (creator_id, consumer_id) to prevent duplicates
- Service role client used for admin operations (cleanup)
- Supabase Realtime for real-time message broadcasting
- Keep-alive ping every 30 seconds to maintain SSE connection

### Database Schema

**conversations table:**
- id (UUID, primary key)
- creator_id (FK to users)
- consumer_id (FK to users)
- created_at, updated_at timestamps
- Unique constraint on (creator_id, consumer_id)

**messages table:**
- id (UUID, primary key)
- conversation_id (FK to conversations, cascade delete)
- sender_id (FK to users, cascade delete)
- content (TEXT)
- created_at, expires_at, read_at timestamps
- Indexes: conversation_id + created_at DESC, expires_at, sender_id

**RLS Policies:**
- Users can view conversations they're part of
- Users can view messages in their conversations
- Users can send messages in their conversations
- Service role has full access

### API Endpoints

**POST /api/messages**
- Creates new message with automatic 24h expiration
- Auto-creates conversation if doesn't exist
- Validates user authorization
- Updates conversation updated_at timestamp
- Returns created message with metadata

**GET /api/messages?conversation_id={id}**
- Retrieves all messages for a conversation
- Filtered by RLS policies
- Ordered by created_at ascending

**GET /api/messages/stream?conversation_id={id}**
- SSE endpoint for real-time message delivery
- Subscribes to Supabase Realtime channel
- Emits messages on INSERT and UPDATE events
- Keep-alive ping every 30 seconds
- Auto-cleanup on client disconnect

**POST /api/messages/cleanup**
- Deletes expired messages (expires_at < NOW)
- Protected by cron secret authorization
- Returns count of deleted messages
- Can be triggered by external cron service

### Cleanup System
- SQL function: `cleanup_expired_messages()`
- Can be scheduled via pg_cron extension
- API endpoint for external cron services (Vercel Cron, etc.)
- Deletes messages where expires_at < NOW()
