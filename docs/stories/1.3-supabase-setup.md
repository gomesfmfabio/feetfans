## Story Description

Set up the Supabase backend infrastructure for the FeetFans marketplace. This includes creating Supabase projects (dev and prod), configuring database migrations with Supabase CLI, creating the initial database schema with the `users` table, implementing Row Level Security (RLS) policies, integrating the Supabase client library into the monorepo, and creating a health check API endpoint to verify database connectivity.

This story establishes the foundational data layer and backend infrastructure required for all subsequent features (authentication, content management, messaging, etc.).

### Users Table Migration

```sql
-- supabase/migrations/00001_create_users_table.sql

-- Create users table
CREATE TABLE IF NOT EXISTS users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT UNIQUE NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('creator', 'consumer')),
  age_verified BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  nickname TEXT,
  is_admin BOOLEAN DEFAULT false
);

-- Create indexes
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_role ON users(role);
CREATE INDEX idx_users_age_verified ON users(age_verified);

-- Create updated_at trigger
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();

-- Enable Row Level Security
ALTER TABLE users ENABLE ROW LEVEL SECURITY;

-- RLS Policy: Users can read their own data
CREATE POLICY "Users can read own data"
  ON users
  FOR SELECT
  USING (auth.uid() = id);

-- RLS Policy: Users can update their own data (limited fields)
CREATE POLICY "Users can update own data"
  ON users
  FOR UPDATE
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);

-- RLS Policy: Service role can access all data
CREATE POLICY "Service role has full access"
  ON users
  USING (auth.role() = 'service_role');
```

### Supabase Client Package

```typescript
// packages/database/src/index.ts
import { createClient } from '@supabase/supabase-js';
import type { Database } from './types';

const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL || process.env.SUPABASE_URL || '';
const supabaseAnonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY || process.env.SUPABASE_ANON_KEY || '';

if (!supabaseUrl || !supabaseAnonKey) {
  throw new Error('Missing Supabase environment variables');
}

export const supabase = createClient<Database>(supabaseUrl, supabaseAnonKey);

export type { Database };
```

### Health Check API Endpoint

```typescript
// apps/api/src/routes/health.ts
import { FastifyPluginAsync } from 'fastify';
import { supabase } from '@feetfans/database';

const healthRoute: FastifyPluginAsync = async (fastify) => {
  fastify.get('/health', async (request, reply) => {
    try {
      // Verify database connectivity
      const { data, error } = await supabase.from('users').select('count').limit(1);

      if (error) {
        return reply.status(503).send({
          status: 'error',
          database: 'disconnected',
          error: error.message,
          timestamp: new Date().toISOString(),
        });
      }

      return reply.status(200).send({
        status: 'ok',
        database: 'connected',
        timestamp: new Date().toISOString(),
      });
    } catch (err) {
      return reply.status(503).send({
        status: 'error',
        database: 'disconnected',
        error: err instanceof Error ? err.message : 'Unknown error',
        timestamp: new Date().toISOString(),
      });
    }
  });
};

export default healthRoute;
```

## Implementation Steps

1. **Install Supabase CLI:**
   ```bash
   npm install -g supabase
   supabase --version
   ```

2. **Create Supabase projects on Supabase Cloud:**
   - Go to https://app.supabase.com
   - Create new project (dev): `feetfans-dev`
   - Create new project (prod): `feetfans-prod` (or defer to later)
   - Copy connection strings and API keys

3. **Initialize Supabase in project:**
   ```bash
   cd /Users/martinez/pegasus
   supabase init
   ```

4. **Link local Supabase to cloud project:**
   ```bash
   supabase login
   supabase link --project-ref <project-ref-from-dashboard>
   ```

5. **Create users table migration:**
   ```bash
   supabase migration new create_users_table
   # Edit generated file in supabase/migrations/
   ```

6. **Apply migration to local Supabase:**
   ```bash
   supabase start
   supabase db reset # Apply all migrations
   ```

7. **Generate TypeScript types from schema:**
   ```bash
   supabase gen types typescript --local > packages/database/src/types.ts
   ```

8. **Create database package:**
   ```bash
   mkdir -p packages/database/src
   cd packages/database
   pnpm init
   pnpm add @supabase/supabase-js
   pnpm add -D typescript
   # Create src/index.ts with Supabase client
   ```

9. **Update .env.example and .env.local:**
   ```
   NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon-key>
   SUPABASE_SERVICE_ROLE_KEY=<service-role-key>
   DATABASE_URL=postgresql://postgres:[password]@db.<project-ref>.supabase.co:5432/postgres
   ```

10. **Create health check API endpoint:**
    - Set up Fastify in `apps/api/src/index.ts`
    - Create `apps/api/src/routes/health.ts`
    - Register route in Fastify app

11. **Test health check locally:**
    ```bash
    cd apps/api
    pnpm dev
    curl http://localhost:3001/api/health
    ```

12. **Push migration to Supabase Cloud:**
    ```bash
    supabase db push
    ```

13. **Verify RLS policies in Supabase Dashboard:**
    - Go to Table Editor > users
    - Check "Row Level Security" is enabled
    - Review policies in "Policies" tab

14. **Commit changes:**
    ```bash
    git add .
    git commit -m "feat: add Supabase backend with users table and RLS [Story 1.3]"
    ```

## Risks & Mitigations

**Risk:** RLS policies might be too restrictive and block legitimate backend operations
**Mitigation:** Use service role key for backend operations that need full access, test policies thoroughly with different user roles

**Risk:** Local Supabase might not match cloud Supabase schema
**Mitigation:** Always apply migrations to local first, test thoroughly, then push to cloud with `supabase db push`

**Risk:** Health check endpoint might expose sensitive database information
**Mitigation:** Return minimal information (just status and timestamp), no query details or error stack traces

**Risk:** Supabase CLI version mismatch between local and cloud
**Mitigation:** Pin Supabase CLI version in package.json, document required version in README
